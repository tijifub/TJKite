# TJKite Admin

Kite school manager. Runs locally as a tiny Node server, deploys to Cloudflare Pages.

## Single source of truth

The HTML, students-data.js and voucher-bg.jpg live at the project root — always edit those, never the copies under `public/`.

`public/` is generated by `sync-public.js`, which copies (not symlinks) the root files in before every `dev:cf` and `deploy` run. Cloudflare Pages / Wrangler doesn't follow symlinks ([cloudflare/workers-sdk#3094](https://github.com/cloudflare/workers-sdk/issues/3094)) and also can't see files outside whatever "Root directory" you configure, which is why `functions/`, `wrangler.toml`, and `public/` all live at the repo root rather than nested in a `cloudflare/` subfolder. The generated files under `public/` are gitignored.

```
.
├── kitesurf-school.html       ← edit here (single source)
├── students-data.js
├── voucher-bg.jpg
├── sync-public.js             ← copies the three files above into public/
├── server.js                  ← local dev server (unchanged)
├── shared-data.json           ← local state (gitignored)
├── package.json
├── public/                    ← generated (gitignored) — do not edit directly
├── functions/
│   ├── _middleware.js         ← auth gate (currently disabled)
│   └── api/
│       ├── state.js           ← KV-backed
│       └── tide.js            ← RWS + Open-Meteo
├── wrangler.toml
├── .dev.vars.example
├── CLOUDFLARE.md              ← deploy details
└── .github/workflows/deploy.yml
```

## Local development

Two ways to run, pick whichever feels closer to prod:

```sh
# Plain Node (no auth, file-based state) — same as before:
npm run dev
# → http://localhost:8787

# Cloudflare runtime locally (no auth currently, KV-backed state):
cp .dev.vars.example .dev.vars   # set TJKITE_PASSWORD locally (gitignored, never commit)
npm run dev:cf   # runs sync-public.js automatically first
```

## CI/CD

Pick one of these:

### Option A — Cloudflare's git integration (simplest, recommended)

1. Push this project to GitHub.
2. Cloudflare dashboard → Workers & Pages → Create → Pages → Connect to Git.
3. Pick the repo. Build settings:
   - **Root directory:** *(leave blank — repo root)*
   - **Build command:** `node sync-public.js`
   - **Build output directory:** `public`
4. Add the `TJKITE_PASSWORD` secret under Settings → Variables and secrets (type: Secret, so it stays encrypted and out of git).
5. Run `npm run kv:create` once locally and paste both ids into `wrangler.toml`. Commit.
6. Every push to `main` deploys automatically. No GitHub secrets, no workflow file needed.

### Option B — GitHub Actions (more control)

The repo ships with `.github/workflows/deploy.yml`. To use it:

1. Create a Cloudflare API token: https://dash.cloudflare.com/profile/api-tokens → "Edit Cloudflare Workers" template (covers Pages too).
2. Find your account id: any Cloudflare dashboard page, right sidebar.
3. In GitHub repo → Settings → Secrets and variables → Actions, add:
   - `CLOUDFLARE_API_TOKEN`
   - `CLOUDFLARE_ACCOUNT_ID`
4. Run `npm run kv:create` once locally, paste ids into `wrangler.toml`, commit.
5. Run `npm run secret:set` once to set `TJKITE_PASSWORD` in Cloudflare (production).
6. Push to `main` → workflow deploys. The workflow runs `sync-public.js` before deploying.

Pick A if you don't need extra steps (tests, lint, multiple environments) — it's zero config beyond pointing Cloudflare at the repo and setting the build command above. Use B if you want the deploy in your own pipeline.

## First-time deploy (manual)

If you'd rather skip CI for the first run:

```sh
npm i
npx wrangler login
npm run kv:create        # paste the two ids into wrangler.toml
npm run secret:set       # set TJKITE_PASSWORD on Cloudflare
npm run deploy           # runs sync-public.js automatically first
```

See `CLOUDFLARE.md` for details (KV migration, free-tier limits, debug tips).

## Auth model

- **Local Node server**: no auth (same as before).
- **Cloudflare**: currently **no auth** — disabled for the test/demo phase (see `functions/_middleware.js` to re-enable HTTP Basic Auth before adding real data). The in-app login screen is client-side only and does not protect `/api/state`.

## Migrating local state to production

```sh
wrangler kv key put --binding=TJKITE_KV tjkite-state "$(cat shared-data.json)" --remote
```
